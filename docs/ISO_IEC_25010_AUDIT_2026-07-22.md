# ISO/IEC 25010 Software Quality Audit

> Audit date: 2026-07-22  
> Scope: Casa Paraiso mobile-first application and Laravel fallback, assessed against the ISO/IEC 25010 evaluation criteria in `output/markdown/Copy of Sprint Deliverable 4.md`.  
> Assessment type: evidence-based codebase audit; it is not an independent certification or a substitute for panel testing with stakeholders.

## Executive Result

The application is **adequately addressed** against the SD4 rubric, with a provisional score of **49/60**. Core workflows, role boundaries, scheduling safeguards, mobile build quality, accessible interaction patterns, and baseline security controls have strong code and test evidence. The main release blockers are production verification, data-recovery acceptance, and a non-green Laravel suite caused by test-configuration drift.

## Detailed Evaluation

### Functional Suitability - 14/15

**Evidence supporting the score**

- The Laravel and mobile workspaces cover the approved operational roles and central workflows: booking, availability, staff assignment, payments, feedback, customer rewards, commissions, reporting, and role-scoped administration.
- `tests/Feature/CalendarSchedulingTest.php` covers customer ownership, role access, booking lead time, midnight-ending availability, overlap prevention, capacity reservation, and protected schedule changes.
- `tests/Feature/RoleAccessTest.php` verifies role isolation and inactive-account logout behavior.
- Form validation, transaction-based appointment workflow services, and success/error states are documented in `docs/PROJECT_MEMORY.md` and exercised by role-specific workflow suites.

**Residual evidence gap**

- The fresh Laravel suite is not green. `MobileCustomerFeedbackApiTest` expected a negative classification for a negative cleanliness comment but received positive, while `SentimentUpgradeTest` received `model-v1.0.0` where the test expects the rules-authoritative `2.0.0` result. These failures are consistent with a test runtime using model mode instead of the test's expected shadow/rules-authoritative configuration.

### Performance Efficiency - 4/5

**Evidence supporting the score**

- `tests/Feature/NavigationPerformanceTest.php` enforces a maximum of 15 database queries for key authenticated dashboard and list routes.
- `tests/Feature/CalendarSchedulingTest.php` requires monthly availability resolution to remain below 20 queries.
- `app/Http/Middleware/MeasureApiRequest.php` records request duration, database duration, query count, and a request ID; it returns `Server-Timing` and `X-Request-ID` headers.
- `mobile/src/lib/mobileDataCache.ts` provides TTL-based cache reuse, concurrent-read de-duplication, invalidation, and delayed preloading.

**Residual evidence gap**

- No current load test, device-network profile, or defined response-time service-level objective was reviewed. The mobile production build also contains a main JavaScript bundle of approximately 457 KB (142 KB gzip), which should be measured on target devices before release.

### Compatibility - 13/15

**Evidence supporting the score**

- The target mobile client is a bundled Vue 3/TypeScript Capacitor Android application; `mobile/package.json` declares Capacitor Android 8 and the production build succeeds.
- The Laravel fallback is server-rendered and its shared UI contract requires responsive list controls, keyboard-focusable overflow, and a compact layout at smaller breakpoints.
- The project memory records prior phone-sized browser and Android-emulator acceptance of the mobile workspaces.

**Residual evidence gap**

- This audit did not re-run a cross-browser matrix, tablet verification, or physical-phone acceptance. Those checks are still needed for the exact release build and hosted endpoint.

### Interaction Capability (Usability) - 9/10

**Evidence supporting the score**

- The mobile and Blade UI use shared role navigation, visible keyboard focus styles, semantic design tokens, consistent controls, and responsive forms/lists.
- `resources/css/app.css` and `mobile/src/style.css` contain `:focus-visible` treatments.
- `tests/Feature/CompactWorkspacePaginationTest.php` verifies fixed paging, retained filter/sort state, clear empty states, active filter disclosure, and pagination ARIA attributes.
- `tests/Feature/ModalInfrastructureTest.php` verifies focus trapping, a centralized modal layer, keyboard tab handling, and scroll locking.
- The SD3 service-oriented form rates all five statements at 4/5, which is positive validation evidence but not a replacement for a complete scored SUS analysis.

**Residual evidence gap**

- The audit did not run the Playwright accessibility suite or conduct a fresh task-based usability session. Record the SUS participant count, scoring calculation, and evaluation summary in the executive portfolio.

### Reliability - 4/5

**Evidence supporting the score**

- Booking and appointment lifecycle behavior is guarded by transactional workflow services and tested for capacity conflicts, invalid schedule changes, terminal-state restrictions, and ownership boundaries.
- The project uses Laravel migrations, isolated test-database conventions, and explicit recovery from invalid input through validation and error responses.
- The mobile cache preserves already loaded content during refresh failures, as documented in project memory.

**Residual evidence gap**

- The Laravel suite completed with 3 failures. In addition to the two sentiment failures, `MobilePairingApiTest` expected `['password']` but received `['password', 'google']`, indicating that live Google-provider configuration is leaking into a test that expects the password-only contract. Test bootstrap/configuration must set deterministic Google and sentiment-mode defaults before the suite can serve as release evidence.
- A non-production restore rehearsal and an agreed backup/export-retention plan remain open in `docs/SECURITY_HARDENING.md`.

### Security - 7/10

**Evidence supporting the score**

- `app/Http/Middleware/AddSecurityHeaders.php` supplies `nosniff`, frame, referrer, permissions, and opener-policy headers, with HSTS gated to secure requests when enabled.
- `tests/Feature/SecurityHardeningTest.php` covers the header baseline, HSTS behavior, and sensitive-route rate-limit middleware.
- `app/Providers/AppServiceProvider.php` defines rate limits for sensitive browser routes and mobile metadata/login/Google endpoints, and can force HTTPS in production.
- `config/session.php` supports database sessions, HTTP-only cookies, SameSite protection, and environment-controlled secure cookies/encryption.
- The Android client uses Capacitor secure storage for native session and Google-auth state (`mobile/src/lib/session.ts`, `mobile/src/lib/googleAuth.ts`).
- The project documents least-privilege Supabase roles, private-schema access, TLS verification, and disabled Data API controls.

**Findings and release blockers**

1. **P1 - Production security and recovery gates are not accepted.** The security checklist still marks production secret configuration, hosted HTTPS/login/logout verification, error-page review, backup retention, restore rehearsal, release-account review, and production security-flow verification as pending. Do not treat the current build as production-ready until these are completed and recorded.
2. **P2 - Terms and consent are not evidenced in the current UI source.** `resources/views/privacy-policy.blade.php` provides a privacy policy, but the audit found no separate Terms and Conditions content or explicit user-consent flow in `routes`, `resources`, or `mobile/src`. This conflicts with the checked “Terms and Conditions” and “User Consent” entries in `docs/SECURITY_HARDENING.md`. Reconcile the checklist with the implementation: either implement and test those requirements on both web and mobile surfaces, or mark them pending with an approved scope decision.
3. **P2 - A strict Content Security Policy remains deferred.** The existing header baseline is useful, but `docs/SECURITY_HARDENING.md` explicitly defers CSP until inline behavior is made nonce-compatible and regression-tested. Track this as a post-MVP hardening item with an owner and acceptance test.

## Recommended Release Order

1. Make the test bootstrap deterministic for Google-provider availability and sentiment classifier mode, then rerun the complete Sail Laravel test suite until it is green. Run the asset build, view-cache check, and formatter check listed in `docs/SECURITY_HARDENING.md`.
2. Resolve the Terms and Conditions / user-consent discrepancy, including routes, visible links or acknowledgement behavior, and web/mobile tests.
3. Complete the production environment, HTTPS, auth-flow, and error-page checks using the approved hosted endpoint.
4. Agree backup retention and perform a documented non-production restore rehearsal.
5. Re-run Android emulator or physical-device and cross-browser acceptance for the final signed APK and hosted build; capture results in the SD4 evaluation summary.
6. Add a CSP migration plan and regression coverage before considering the security characteristic fully addressed.

## Audit Conclusion

The current codebase provides a credible MVP for the SD4 functional, usability, and operational requirements. It should be presented as **adequately addressed**, with the explicit condition that the Laravel suite is restored to green and the P1 production/recovery gates and P2 privacy-consent discrepancy are closed before production deployment or a final “fully addressed” security claim.
